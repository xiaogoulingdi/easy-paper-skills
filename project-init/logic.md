# Project Init 逻辑说明

本文档用于解释 `project-init` 模块的工作方式，以及它与其他模块之间的协作关系。

## 核心使命

`project-init` 的目标是先生成一份全局统一配置，让后续所有模块在同一个研究边界内执行任务，避免写作和评审阶段出现术语漂移或方向偏题。

核心产物如下：

- 工作区根目录下的 `USER_RESEARCH_PROFILE.md`

## 为什么必须先初始化

如果没有统一配置，不同 Agent 可能会采用不同假设、不同术语和不同评判标准，导致：

- 研究阶段与写作阶段不一致
- 正文与摘要术语不统一
- 评审时出现“标准错位”

完成初始化后，将获得以下收益：

- 统一的领域边界
- 统一的术语契约
- 统一的体裁约束

## 与其他模块的联动流程

1. `project-init` 收集课题与约束条件，生成或更新配置文件。
2. `deep-research` 读取配置文件，约束检索范围、关键词和综合报告措辞。
3. `academic-paper` 读取配置文件，约束大纲结构、正文聚焦点和写作术语。
4. `academic-paper-reviewer` 读取配置文件，按照同一标准进行方法学与逻辑审查。

## 上下文节省策略

本模块的设计目标之一是减少上下文消耗：

- 初始化阶段只加载 `project-init/SKILL.md` 及其必要资产
- 不默认读取其他模块全部技能文件
- 初始化完成后，其他模块仅需读取“本模块文件 + 配置文件”即可工作

## 最低必填字段

- 课题主题
- 学科与细分方向
- 论文体裁
- 核心方法
- 关键评价指标
- 术语表
- 红线约束

## 推荐使用生命周期

- 第 0 天：先运行 `project-init`
- 后续写作期：按需调用 `deep-research`、`academic-paper`、`academic-paper-reviewer`
- 题目或方法有重大变化：重新运行 `project-init` 更新配置
